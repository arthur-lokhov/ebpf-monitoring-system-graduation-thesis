# Техническая документация eBPF Monitoring Platform

## ВВЕДЕНИЕ

В современных распределённых информационных системах обеспечение надёжного мониторинга и наблюдения за производительностью является критически важным фактором. С ростом сложности инфраструктуры микросервисов и облачных приложений традиционные методы мониторинга на уровне пользовательского пространства не обеспечивают достаточной глубины наблюдения за системными процессами.

Технология eBPF (extended Berkeley Packet Filter) представляет собой революционный подход к наблюдению за операционной системой Linux, позволяющий выполнять произвольный код в пространстве ядра без внесения изменений в его исходный код. Это обеспечивает беспрецедентную видимость системных событий при минимальном влиянии на производительность.

Актуальность данной работы обусловлена возрастающей потребностью организаций в эффективных инструментах мониторинга, способных работать на уровне ядра операционной системы, обеспечивая при этом безопасность, изоляцию и возможность расширения функциональности через систему плагинов.

Целью данной практической работы является проектирование и разработка модульной платформы мониторинга на основе eBPF с поддержкой пользовательских плагинов, распространяемых через Git-репозитории, с возможностью гибкой фильтрации и визуализации метрик в реальном времени.

### 1.2 Физическая модель базы данных

Физическая модель данных в контексте СУБД представляет собой детализированное описание того, как данные будут храниться и организовываться внутри конкретной базы данных, учитывая спецификации и характеристики PostgreSQL 15. Она определяет структуру таблиц, индексы, типы данных, размеры полей, механизмы хранения, а также правила для обеспечения целостности данных и оптимизации запросов.

**Реализованные таблицы:**

**1) plugins (Плагины):**
- `id` UUID PRIMARY KEY DEFAULT gen_random_uuid() — уникальный идентификатор плагина;
- `name` VARCHAR(255) NOT NULL UNIQUE — имя плагина;
- `version` VARCHAR(50) NOT NULL — версия плагина;
- `description` TEXT — описание плагина;
- `author` VARCHAR(255) — автор плагина;
- `git_url` TEXT NOT NULL — URL Git-репозитория;
- `git_commit` VARCHAR(40) — хеш коммита;
- `git_branch` VARCHAR(255) DEFAULT 'main' — ветка репозитория;
- `ebpf_s3_key` TEXT — ключ eBPF артефакта в S3;
- `wasm_s3_key` TEXT — ключ WASM модуля в S3 (резерв);
- `manifest` JSONB NOT NULL — манифест плагина в формате JSON;
- `status` VARCHAR(50) DEFAULT 'pending' — статус (pending, building, ready, error);
- `build_log` TEXT — лог сборки;
- `error_message` TEXT — сообщение об ошибке;
- `created_at` TIMESTAMP WITH TIME ZONE DEFAULT NOW();
- `updated_at` TIMESTAMP WITH TIME ZONE DEFAULT NOW();

**Индексы:**
- `idx_plugins_status` — для фильтрации плагинов по статусу;
- `idx_plugins_name` — для быстрого поиска по имени;
- `idx_plugins_created_at` — для сортировки по дате создания.

**2) metrics (Метрики):**
- `id` UUID PRIMARY KEY DEFAULT gen_random_uuid() — уникальный идентификатор метрики;
- `plugin_id` UUID NOT NULL REFERENCES plugins(id) ON DELETE CASCADE — плагин-владелец;
- `name` VARCHAR(255) NOT NULL — имя метрики;
- `type` VARCHAR(50) NOT NULL — тип (counter, gauge, histogram, summary);
- `help` TEXT — описание метрики;
- `labels` JSONB DEFAULT '[]'::jsonb — доступные лейблы метрики;
- `created_at` TIMESTAMP WITH TIME ZONE DEFAULT NOW();
- UNIQUE(plugin_id, name);

**Индексы:**
- `idx_metrics_plugin_id` — для группировки метрик по плагинам;
- `idx_metrics_name` — для быстрого поиска по имени;
- `idx_metrics_type` — для фильтрации по типу метрики.

**3) filters (Фильтры метрик):**
- `id` UUID PRIMARY KEY DEFAULT gen_random_uuid() — уникальный идентификатор фильтра;
- `plugin_id` UUID REFERENCES plugins(id) ON DELETE CASCADE — плагин-владелец;
- `name` VARCHAR(255) NOT NULL — имя фильтра;
- `expression` TEXT NOT NULL — выражение фильтра (PromQL-подобный DSL);
- `description` TEXT — описание назначения фильтра;
- `is_default` BOOLEAN DEFAULT false — является ли фильтром по умолчанию;
- `created_at` TIMESTAMP WITH TIME ZONE DEFAULT NOW();
- UNIQUE(plugin_id, name);

**Индексы:**
- `idx_filters_plugin_id` — для группировки фильтров по плагинам;
- `idx_filters_name` — для быстрого поиска по имени.

**4) dashboards (Дашборды):**
- `id` UUID PRIMARY KEY DEFAULT gen_random_uuid() — уникальный идентификатор дашборда;
- `name` VARCHAR(255) NOT NULL — имя дашборда;
- `description` TEXT — описание дашборда;
- `config` JSONB NOT NULL — конфигурация дашборда (панели, макет);
- `is_default` BOOLEAN DEFAULT false — является ли дашбордом по умолчанию;
- `created_at` TIMESTAMP WITH TIME ZONE DEFAULT NOW();
- `updated_at` TIMESTAMP WITH TIME ZONE DEFAULT NOW();

**Индексы:**
- `idx_dashboards_name` — для поиска по имени.

**5) plugin_events (Журнал событий плагинов):**
- `id` UUID PRIMARY KEY DEFAULT gen_random_uuid() — уникальный идентификатор события;
- `plugin_id` UUID NOT NULL REFERENCES plugins(id) ON DELETE CASCADE — плагин;
- `event_type` VARCHAR(50) NOT NULL — тип события (created, building, ready, error, deleted);
- `message` TEXT — сообщение события;
- `metadata` JSONB DEFAULT '{}'::jsonb — метаданные события;
- `created_at` TIMESTAMP WITH TIME ZONE DEFAULT NOW();

**Индексы:**
- `idx_plugin_events_plugin_id` — для группировки событий по плагинам;
- `idx_plugin_events_event_type` — для фильтрации по типу события;
- `idx_plugin_events_created_at` — для сортировки по времени.

**Связи между таблицами:**

1) **plugins ↔ metrics**: Один плагин генерирует много метрик (One-to-Many).
   - `metrics.plugin_id → plugins.id` (ON DELETE CASCADE)

2) **plugins ↔ filters**: Один плагин может иметь много фильтров (One-to-Many).
   - `filters.plugin_id → plugins.id` (ON DELETE CASCADE)

3) **plugins ↔ plugin_events**: Один плагин имеет множество событий в журнале (One-to-Many).
   - `plugin_events.plugin_id → plugins.id` (ON DELETE CASCADE)

4) **plugins ↔ dashboards**: Прямой связи нет, дашборды агрегируют метрики из разных плагинов через JSONB-конфигурацию.

**Обоснование выбора типов данных:**

- **UUID** для первичных ключей: гарантирует уникальность в распределённых системах, упрощает шардирование в будущем;
- **JSONB** для манифестов, лейблов, конфигураций и метаданных событий: PostgreSQL эффективно индексирует JSONB через GIN-индексы, позволяет выполнять сложные запросы по полуструктурированным данным;
- **TIMESTAMP WITH TIME ZONE** для временных меток: поддержка часовых поясов и интервальных запросов;
- **TEXT** для описаний, логов и сообщений об ошибках: неограниченная длина для произвольного текста.

**Механизмы хранения:**

PostgreSQL использует механизм Heap Tables для хранения данных. Для таблицы `plugin_events`, которая подвержена высокой нагрузке записи, рекомендуется:
- Партиционирование по времени (например, по дням/неделям) при росте объёма данных;
- Регулярная очистка старых данных через TTL-политики.

### 1.3 Схема развертывания аппаратного и программного обеспечения

**Аппаратное обеспечение:**

1) **Серверная часть (Backend):**
   - Сервер с ОС Linux 5.8+ (для поддержки eBPF);
   - Процессор: x86_64 с поддержкой инструкций JIT-компиляции eBPF;
   - Оперативная память: минимум 2 ГБ (рекомендуется 4+ ГБ для production);
   - Дисковое пространство: 10+ ГБ для базы данных и S3-хранилища;
   - Сетевой интерфейс: 1 Гбит/с для приёма метрик и WebSocket-соединений.

2) **Клиентские устройства:**
   - Любой современный браузер (Chrome, Firefox, Edge, Safari);
   - Подключение через интернет или локальную сеть (HTTP/WebSocket).

3) **Инфраструктурные компоненты:**
   - PostgreSQL сервер (в Docker-контейнере или нативно);
   - Garage S3 сервер (в Docker-контейнере);
   - Prometheus сервер для сбора метрик (в Docker-контейнере).

**Программное обеспечение:**

1) **Серверная часть:**
   - Операционная система: Linux (Ubuntu 22.04+, Debian 12+, Fedora 38+);
   - Ядро Linux: 5.8+ (обязательно для eBPF);
   - Go 1.21+ (для компиляции и запуска);
   - Clang 14+ и LLVM (для сборки eBPF-программ);
   - Docker и Docker Compose (для развёртывания зависимостей);
   - Make (для автоматизации сборки).

2) **База данных:**
   - PostgreSQL 15 — реляционная база данных;
   - Расширение pgcrypto (для UUIDv4);
   - Миграции через `migrate` или ручные SQL-скрипты.

3) **Объектное хранилище:**
   - Garage S3 v2.2.0 — S3-совместимое хранилище;
   - Хранение .o (eBPF) файлов плагинов.

4) **Сбор метрик:**
   - Prometheus v2.50.1 — сбор и хранение временных рядов;
   - Конфигурация scrape_configs для polling eBPF-сервера.

5) **Клиентская часть:**
   - Node.js 18+ и npm 9+ (для разработки);
   - React 18 + TypeScript + Vite (для сборки);
   - shadcn/ui компоненты + Tailwind CSS (для стилизации);
   - Recharts (для визуализации метрик).

**Взаимодействие компонентов:**

1. Пользователь открывает веб-приложение (React UI) в браузере на порту 3000.
2. React UI отправляет REST-запросы к Go API серверу на порту 8080 (`/api/v1/plugins`, `/api/v1/metrics`, `/api/v1/filters`, `/api/v1/dashboard`).
3. Для обновлений в реальном времени UI устанавливает WebSocket-соединение (`ws://localhost:8080/ws`).
4. Go API сервер обрабатывает запросы:
   - Для операций с плагинами: взаимодействует с PostgreSQL (метаданные) и Garage S3 (бинарные eBPF файлы);
   - Для метрик: читает данные из памяти, выполняет filter engine запросы, возвращает результаты;
   - Для фильтров и дашбордов: CRUD-операции через PostgreSQL.
5. eBPF Loader загружает eBPF-программы в ядро Linux через `cilium/ebpf` библиотеку.
6. Prometheus опрашивает endpoint `/metrics` на Go сервере каждые 10-15 секунд.
7. Grafana (опционально) подключается к Prometheus для сложной визуализации через Grafana Scenes.

**Развёртывание через Docker Compose:**

Проект использует `docker-compose.yml` для локального развёртывания:

```yaml
services:
  ui:              # React приложение (порт 3000)
  prometheus:      # Сбор метрик (порт 9090)
  postgres:        # Метаданные (порт 5432)
  garage:          # S3-хранилище (порт 3900)
  epbf-monitor:    # Go сервер (запускается на хосте для доступа к eBPF)
```

**Требования к безопасности:**

- eBPF программы проходят верификацию ядром Linux (eBPF verifier);
- Плагины компилируются в Docker-контейнерах с ограничением ресурсов (512MB RAM, 1 CPU);
- Rootless режим — работа без root-прав благодаря проверке eBPF verifier'ом;
- Ограниченные capabilities: только `CAP_BPF` и `CAP_PERFMON`.

## ЗАКЛЮЧЕНИЕ

В ходе работы над проектом была спроектирована и разработана модульная платформа мониторинга eBPF Monitoring Platform, которая предоставляет пользователям возможность управлять eBPF-плагинами, просматривать метрики в реальном времени и настраивать визуализацию через гибкую систему дашбордов.

Актуальность проекта обусловлена возрастающей потребностью организаций в инструментах мониторинга, способных работать на уровне ядра операционной системы, обеспечивая при этом высокую производительность, безопасность и возможность расширения через систему плагинов.

Анализ существующих решений позволил выявить их сильные и слабые стороны, что дало возможность учесть полученные выводы при проектировании системы. В ходе работы были определены ключевые функциональные и нефункциональные требования, спроектирована архитектура приложения, выбран технологический стек.

Тестирование приложения подтвердило его работоспособность, удобство использования и надёжность системы. Архитектура на основе eBPF обеспечивает глубокую видимость системных процессов при минимальном влиянии на производительность, а система плагинов позволяет гибко расширять функциональность мониторинга.

**Основные результаты работы:**

1. Разработан полный технологический стек: Go (backend), React/TypeScript (frontend), PostgreSQL (база данных), Garage S3 (хранилище), Prometheus (метрики);

3. Разработана схема развёртывания через Docker Compose с поддержкой локального тестирования и production-готовности;
4. Создан современный веб-интерфейс с четырьмя основными страницами (Plugins, Metrics, Dashboard, Filters) и WebSocket для обновлений в реальном времени;
5. Реализован PromQL-подобный DSL для фильтрации и агрегации метрик (rate, sum, avg, histogram_quantile);
6. Внедрена система сборки плагинов в Docker-контейнерах с загрузкой eBPF-программ в ядро Linux;
7. Реализован event-driven механизм обработки eBPF-событий с маппингом на метрики;
8. Проведено функциональное тестирование, подтвердившее работоспособность всех компонентов.

Практическая значимость проекта заключается в создании удобной и функциональной платформы мониторинга, позволяющей системным администраторам и DevOps-инженерам получать глубокую видимость системных процессов через eBPF, не влияя на производительность серверов.
