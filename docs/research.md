# 1 ИССЛЕДОВАТЕЛЬСКИЙ РАЗДЕЛ

## 1.1 Характеристика предметной области

Мониторинг ИТ-инфраструктуры — это процесс непрерывного наблюдения за состоянием вычислительных систем, сетей и приложений с целью обеспечения их доступности, производительности и безопасности [1]. В условиях цифровой трансформации бизнеса системы мониторинга становятся критически важным компонентом ИТ-ландшафта, позволяющим предотвращать простои, выявлять аномалии и оптимизировать использование ресурсов [2].

Современные распределенные системы и микросервисные архитектуры порождают новые вызовы для традиционных подходов к мониторингу. Контейнеризация приложений, динамическое масштабирование и оркестрация создают высокую степень изменчивости инфраструктуры, что требует инструментов наблюдения с минимальными накладными расходами и максимальной детализацией данных [3].

В настоящее время специалисты по эксплуатации и разработке сталкиваются со следующими проблемами при организации мониторинга [4]:

1. **Высокие накладные расходы традиционных агентов**. Классические системы мониторинга используют опрос метрик через фиксированные интервалы времени (polling), что создает дополнительную нагрузку на систему. При увеличении частоты опроса для повышения детализации данных пропорционально растут и затраты ресурсов.

2. **Недостаточная видимость на уровне ядра**. Большинство инструментов собирают метрики на уровне приложения или операционной системы, не предоставляя доступа к низкоуровневым событиям: системным вызовам, сетевым пакетам, операциям ввода-вывода. Это затрудняет диагностику сложных проблем производительности.

3. **Отсутствие гибкости кастомизации**. Коммерческие решения предлагают ограниченный набор встроенных метрик, а добавление пользовательских сборщиков данных требует модификации исходного кода или сложной конфигурации.

4. **Фрагментация инструментов**. Различные команды используют разрозненные решения для мониторинга инфраструктуры, приложений и безопасности, что усложняет корреляцию событий и формирование единой картины состояния системы.

5. **Задержки в обнаружении инцидентов**. Интервальный опрос метрик создает временное окно между возникновением проблемы и её обнаружением. Для критически важных систем даже минуты простоя могут означать существенные финансовые потери.

6. **Невозможность мониторинга закрытых систем**. Традиционные инструменты требуют наличия экспортеров метрик или доступа к исходному коду для инструментирования. Проприетарное ПО, скомпилированные бинарные файлы и сторонние сервисы остаются «черными ящиками», недоступными для детального наблюдения.

Технология eBPF (extended Berkeley Packet Filter) представляет собой инновационный подход к решению перечисленных проблем. eBPF позволяет выполнять пользовательский код непосредственно в ядре операционной системы Linux без необходимости загрузки сторонних модулей ядра [5].

**Архитектура eBPF**

eBPF состоит из нескольких ключевых компонентов [6]:

1. **Виртуальная машина eBPF** — изолированная среда исполнения в ядре Linux с собственной системой команд (аналогично байт-коду Java). VM eBPF гарантирует, что программы не могут повредить ядро или другие процессы.

2. **Верификатор** — модуль ядра, который проверяет каждую eBPF-программу перед запуском на предмет:
   - Отсутствия бесконечных циклов (все циклы должны иметь ограниченное число итераций)
   - Безопасных обращений к памяти (нет выхода за границы выделенной области)
   - Допустимых системных вызовов и функций ядра
   - Ограниченной сложности программы (максимум 1 миллион инструкций)

3. **JIT-компилятор** — преобразует байт-код eBPF в нативный машинный код для целевой архитектуры (x86_64, ARM64), что обеспечивает высокую производительность исполнения.

4. **Карты (maps)** — структуры данных для хранения состояния между вызовами eBPF-программ. Типы карт включают:
   - Hash maps — для хранения пар ключ-значение
   - Array maps — для хранения массивов
   - Ring buffers — для эффективной передачи событий в пользовательское пространство
   - LRU maps — для кэширования с вытеснением наименее используемых элементов

5. **Хелперы (helpers)** — набор функций ядра, доступных eBPF-программам для безопасного взаимодействия с системой (отправка событий, работа с картами, получение времени и т.д.).

**Типы eBPF-программ**

eBPF-программы классифицируются по типу точки подключения [7]:

1. **BPF_PROG_TYPE_KPROBE** — прикрепляются к функциям ядра через механизм kprobe. Позволяют отслеживать вход и выход функций ядра, анализировать параметры.

2. **BPF_PROG_TYPE_TRACEPOINT** — прикрепляются к статическим точкам трассировки в ядре. В отличие от kprobe, гарантируют стабильность интерфейса между версиями ядра.

3. **BPF_PROG_TYPE_XDP** — обрабатывают сетевые пакеты на самой ранней стадии, до выделения структур ядра sk_buff. Используются для фильтрации, маршрутизации и анализа трафика.

4. **BPF_PROG_TYPE_SOCKET_FILTER** — фильтруют события на уровне сокетов, позволяя отслеживать сетевые соединения процессов.

5. **BPF_PROG_TYPE_PERF_EVENT** — реагируют на события производительности процессора (аппаратные счётчики, программные события).

6. **BPF_PROG_TYPE_CGROUP_DEVICE** — контролируют доступ к устройствам внутри cgroup.

**Процесс исполнения eBPF-программы**

Жизненный цикл eBPF-программы включает следующие этапы [8]:

1. **Компиляция** — исходный код на C/Rust компилируется в ELF-объект с байт-кодом eBPF и BTF-информацией (BPF Type Format для отладки).

2. **Загрузка** — пользовательское приложение загружает ELF-объект в ядро через системный вызов `bpf(BPF_PROG_LOAD, ...)`.

3. **Верификация** — верификатор ядра анализирует программу на безопасность. При ошибке загрузка отклоняется.

4. **Прикрепление** — программа прикрепляется к точке подключения (kprobe, tracepoint, XDP и т.д.) через link.

5. **Исполнение** — программа исполняется при наступлении соответствующего события в ядре.

6. **Сбор данных** — результаты исполнения передаются в пользовательское пространство через ring buffer или perf event.

**Безопасность eBPF**

Архитектура eBPF обеспечивает многоуровневую защиту [9]:

- **Статическая верификация** — все программы проверяются перед запуском
- **Изоляция памяти** — программы не могут обращаться к произвольной памяти ядра
- **Ограничение ресурсов** — лимиты на количество инструкций, размер карт, время исполнения
- **Capability-модель** — для загрузки программ требуются привилегии CAP_BPF, CAP_PERFMON, CAP_NET_ADMIN
- **Аудит** — все загрузки и события логируются через audit subsystem

**Преимущества eBPF для мониторинга**

- **Минимальные накладные расходы**. Программы eBPF исполняются непосредственно в ядре, обрабатывая события в момент их возникновения. Это исключает необходимость опроса и снижает нагрузку на процессор.
- **Глубокая видимость**. eBPF предоставляет доступ к широкому спектру событий ядра: сетевые соединения, системные вызовы, операции с файлами, использование памяти, планирование процессов.
- **Гибкость**. Разработчики могут создавать пользовательские программы для сбора специфических метрик без модификации ядра или перезагрузки системы.
- **Мониторинг «черных ящиков»**. eBPF отслеживает события на уровне ядра, что позволяет получать детальные метрики о работе любых приложений, включая проприетарное ПО и скомпилированные бинарные файлы, без доступа к их исходному коду и без необходимости инструментирования [10].

Рынок observability-решений демонстрирует устойчивый рост: согласно отчету Gartner, сегмент инструментов мониторинга и управления производительностью приложений достигнет 16,3 миллиарда долларов к 2027 году [11]. eBPF занимает всё большую долю этого рынка, становясь стандартом де-факто для мониторинга облачных инфраструктур [12].

Для решения выявленных проблем и реализации преимуществ eBPF целесообразно разработать платформу мониторинга с модульной архитектурой. Такая система должна предоставлять:

- **Систему плагинов** — механизм создания и загрузки пользовательских сборщиков метрик на основе eBPF.
- **Интеграцию с Prometheus** — экспорт собранных данных в формате, совместимом с экосистемой Prometheus [11].
- **Реестр плагинов** — централизованное хранилище для распространения и обмена расширениями.
- **Веб-интерфейс** — инструмент управления плагинами и визуализации метрик в реальном времени.

Таким образом, разработка eBPF-платформы мониторинга с поддержкой плагинов представляет собой актуальное решение, способное устранить недостатки существующих инструментов и предоставить гибкий механизм кастомизации под специфические требования организаций. Возможность мониторинга закрытых систем без доступа к исходному коду делает платформу универсальным инструментом для гетерогенных инфраструктур.

## 1.2 Входные и выходные данные

Моделирование данных является фундаментальным этапом проектирования информационной системы. Корректное определение входных и выходных данных позволяет установить границы системы, выявить необходимые интерфейсы и спроектировать структуру хранения информации [12].

В контексте платформы мониторинга на основе eBPF входные данные представляют собой информацию, поступающую в систему из внешних источников: событий ядра, конфигурационных файлов, пользовательских запросов. Выходные данные — это результаты обработки входной информации: метрики, уведомления, визуализации, доступные для потребления внешними системами.

**Входные данные платформы:**

1. **События ядра Linux**:
   - Системные вызовы (syscall)
   - Сетевые события (подключения, отключения, передача данных)
   - События планировщика процессов
   - Операции ввода-вывода
   - События работы с памятью
   - Прерывания и исключения процессора

2. **Конфигурационные данные**:
   - Параметры загрузки плагинов
   - Правила фильтрации событий
   - Настройки экспорта метрик
   - Пороговые значения для уведомлений

3. **Пользовательские команды**:
   - Запросы на загрузку/выгрузку плагинов
   - Команды управления конфигурацией
   - Запросы на получение метрик
   - Подписки на уведомления

4. **Данные плагинов**:
   - Исходный код eBPF-программ
   - Манифесты описания плагинов
   - Параметры настройки сборщиков
   - Версионная информация

**Выходные данные платформы:**

1. **Метрики производительности**:
   - Счётчики событий (количество системных вызовов, сетевых подключений)
   - Гистограммы распределения (время выполнения операций, задержки)
   - Gauge-метрики (текущее использование ресурсов)
   - Сводные статистические данные

2. **Данные для экспорта**:
   - Форматированные метрики Prometheus
   - JSON-представления событий
   - Агрегированные отчёты за период

3. **Уведомления и алерты**:
   - События превышения пороговых значений
   - Сообщения об ошибках загрузки плагинов
   - Предупреждения о недоступности источников данных

4. **Визуализации**:
   - Графики временных рядов
   - Диаграммы распределения
   - Топологии сетевых соединений
   - Дашборды состояния системы

5. **Данные реестра плагинов**:
   - Каталоги доступных плагинов
   - Рейтинги и отзывы пользователей
   - Статистика загрузок
   - Информация о совместимости

Поток данных в системе можно представить следующим образом: события ядра поступают в eBPF-программы плагинов, где обрабатываются и агрегируются. Результаты передаются в пользовательское пространство, где формируются в метрики и экспортируются через Prometheus или визуализируются в веб-интерфейсе.

## 1.3 Модель системы "AS-IS"

Для корректного определения требований к разрабатываемой системе необходимо проанализировать существующие подходы к мониторингу инфраструктуры. Построим функциональную модель бизнес-процессов "as-is" в нотации IDEF0, описывающую текущее состояние предметной области.

Основными процессами, протекающими в ходе мониторинга ИТ-инфраструктуры с использованием традиционных инструментов, являются:

1. **Сбор метрик через опрос** — периодический запрос данных о состоянии системы через фиксированные интервалы времени.
2. **Агрегация и хранение данных** — накопление собранных метрик в базах данных временных рядов.
3. **Визуализация и уведомление** — представление данных пользователям и генерация алертов при превышении пороговых значений.

Наиболее распространёнными средствами реализации данных процессов являются системы мониторинга с архитектурой на основе опроса: Prometheus, Zabbix, Nagios, Datadog Agent [13].

Анализ модели "AS-IS" выявляет следующие недостатки текущих подходов:

1. **Циклический опрос создает нагрузку**. Каждый запрос метрик потребляет ресурсы как мониторируемой системы, так и системы мониторинга. При увеличении количества метрик и частоты опроса нагрузка растет линейно.

2. **Задержка обнаружения событий**. Если событие произошло сразу после опроса, оно будет обнаружено только при следующем запросе. Среднее время обнаружения составляет половину интервала опроса.

3. **Потеря кратковременных аномалий**. События, длящиеся меньше интервала опроса, могут остаться незамеченными или искаженными в агрегированных данных.

4. **Ограниченная детализация**. Получение данных на уровне отдельных событий ядра требует экстремально высокой частоты опроса, что делает систему неработоспособной.

5. **Невозможность работы с закрытыми системами**. Традиционные инструменты требуют наличия экспортеров или агентов внутри приложений. Проприетарное ПО остаётся «черным ящиком».

Данные ограничения обуславливают необходимость перехода к модели "TO-BE", основанной на технологии eBPF, где сбор метрик происходит событийно-ориентированным способом: программы активируются только при наступлении соответствующих событий ядра, обеспечивая минимальные накладные расходы и максимальную детализацию данных.

## 1.4 Анализ существующих решений

На основе модели системы "AS-IS" был проведён анализ существующих решений в области мониторинга на основе eBPF. Для проведения сравнения были разработаны критерии оценки, позволяющие определить необходимость создания новой платформы.

### Критерии сравнения

1. **Архитектура на основе eBPF** — использует ли решение технологию eBPF для сбора метрик непосредственно в ядре Linux, обеспечивая минимальные накладные расходы.

   Данный критерий является фундаментальным для оценки современного инструмента мониторинга. eBPF обеспечивает событийно-ориентированный сбор данных, активируя программы только при наступлении соответствующих событий ядра. Это устраняет необходимость циклического опроса и снижает нагрузку на систему до 10–100 раз по сравнению с традиционными агентами [14]. Решение, использующее eBPF, способно предоставлять метрики с детализацией до отдельных системных вызовов без деградации производительности.

2. **Поддержка пользовательских плагинов** — существует ли возможность создания и загрузки собственных сборщиков метрик без модификации базового кода системы.

   Гибкость кастомизации определяет адаптивность платформы к специфическим требованиям организации. Система плагинов позволяет разработчикам создавать специализированные сборщики для мониторинга уникальных аспектов инфраструктуры: от отслеживания конкретных системных вызовов до анализа протоколов прикладного уровня. Возможность динамической загрузки плагинов без перезапуска системы обеспечивает непрерывность мониторинга при расширении функциональности [15].

3. **Интеграция с экосистемой Prometheus** — поддерживается ли экспорт метрик в формате Prometheus для совместимости с существующими системами сбора и визуализации.

   Prometheus де-факто стал стандартом для мониторинга облачных инфраструктур и Kubernetes-кластеров. Интеграция с экосистемой Prometheus (включая совместимость с Grafana, Alertmanager, Thanos) критически важна для внедрения инструмента в существующие процессы эксплуатации. Нативная поддержка формата экспорта Prometheus устраняет необходимость в дополнительных шлюзах и преобразователях данных [16].

4. **Механизм распространения плагинов** — существует ли централизованный реестр для публикации, поиска и загрузки плагинов, созданных сообществом.

   Экосистема плагинов определяет жизнеспособность платформы в долгосрочной перспективе. Централизованный реестр с системой верификации, рейтингами и отзывами стимулирует сообщество к созданию качественных расширений. Отсутствие такого механизма приводит к фрагментации разработок и дублированию усилий [17].

5. **Безопасность исполнения кода** — реализованы ли механизмы валидации и изоляции пользовательских eBPF-программ для предотвращения влияния на стабильность системы.

   Поскольку eBPF-программы исполняются в ядре, критически важно обеспечить их безопасность. Верификатор eBPF проверяет код на предмет бесконечных циклов, недопустимых обращений к памяти и других потенциально опасных конструкций. Дополнительные механизмы изоляции (resource limits, capability restrictions) предотвращают исчерпание ресурсов ядра некорректными плагинами [18].

6. **Мониторинг закрытых систем** — способна ли платформа собирать метрики о работе приложений без доступа к их исходному коду и без необходимости инструментирования.

   Критический критерий для гетерогенных инфраструктур, где значительная часть ПО является проприетарным. Платформа должна позволять создавать плагины для мониторинга любых процессов на уровне системных вызовов и событий ядра, независимо от наличия экспортеров метрик [19].

В результате анализа были выявлены следующие коммерческие и открытые платформы для мониторинга на основе eBPF:

### 1. Коммерческие платформы на основе eBPF

**Cilium/Hubble** [20] — платформа сетевой безопасности и наблюдаемости, разработанная компанией Isovalent. Cilium использует eBPF для реализации политик сетевого доступа, а Hubble предоставляет визуализацию сетевых событий. Функционал включает:
- Мониторинг сетевых соединений между контейнерами и узлами кластера
- Визуализацию топологии сервисов Kubernetes
- Детектирование аномалий сетевого трафика
- Интеграцию с Prometheus через экспорт метрик
- Поддержку политик безопасности на уровне L3–L7

- Мониторинг на уровне L7 (HTTP, gRPC, Kafka)

Недостатки:
- Узкая специализация на сетевом мониторинге
- Отсутствие поддержки пользовательских плагинов

- Ограниченная возможность мониторинга вне Kubernetes

**Pixie** [21] — платформа наблюдаемости для Kubernetes, приобретенная компанией New Relic. Pixie автоматически инструментализует приложения с помощью eBPF, предоставляя данные о производительности без необходимости изменения кода. Функционал включает:
- Автоматический сбор метрик HTTP, gRPC, PostgreSQL, Redis
- Трассировку распределенных транзакций
- Живые дашборды с метриками в реальном времени
- Скриптовый язык PX для создания пользовательских запросов
- Поддержку мониторинга проприетарных приложений через eBPF

Достоинства:
- Автоматическая инструментализация
- Поддержка множественных протоколов прикладного уровня
- Встроенный язык запросов
- Возможность мониторинга «черных ящиков» через системные вызовы

Недостатки:
- Ограниченность поддержки только Kubernetes
- Отсутствие механизма распространения скриптов
- Зависимость от облачной инфраструктуры New Relic
- Нет полноценной системы плагинов

**Falco** [22] — система обнаружения вторжений на основе eBPF, разработанная Sysdig (приобретена Cisco). Falco отслеживает события ядра и генерирует уведомления при обнаружении подозрительной активности. Функционал включает:
- Мониторинг системных вызовов и сетевых событий
- Правила обнаружения аномалий (YAML-синтаксис)
- Интеграцию с SIEM-системами и системами уведомлений
- Аудит безопасности в реальном времени
- Поддержку мониторинга любых процессов без модификации

Достоинства:
- Фокус на безопасности
- Гибкая система правил
- Интеграция с облачными провайдерами
- Мониторинг закрытых систем через syscall

Недостатки:
- Ориентация только на безопасность, не на производительность
- Отсутствие поддержки пользовательских программ на eBPF
- Нет встроенного экспорта Prometheus
- Нет реестра правил с верификацией

**Datadog eBPF** [23] — коммерческая платформа наблюдаемости от компании Datadog, использующая eBPF для сбора метрик инфраструктуры. Функционал включает:
- Автоматический сбор метрик системных вызовов
- Профилирование приложений без инструментирования
- Сетевой мониторинг на уровне процессов
- Интеграцию с дашбордами Datadog
- Поддержку мониторинга проприетарного ПО

Достоинства:
- Полноценная коммерческая поддержка
- Глубокая интеграция с экосистемой Datadog
- Автоматическое обнаружение сервисов
- Мониторинг любых приложений через eBPF

Недостатки:
- Закрытый исходный код
- Высокая стоимость лицензий
- Отсутствие возможности создания пользовательских плагинов
- Привязка к облаку Datadog

**Dynatrace OneAgent** [24] — платформа наблюдаемости от Dynatrace, использующая eBPF для низкоуровневого сбора метрик. Функционал включает:
- Автоматическую инструментализацию через eBPF
- Мониторинг системных вызовов и сетевых событий
- Машинное обучение для обнаружения аномалий
- Трассировку распределенных транзакций
- Поддержку мониторинга закрытых систем

Достоинства:
- Автоматическая базовая линия производительности
- ИИ для обнаружения аномалий
- Глубокая видимость инфраструктуры
- Мониторинг без доступа к исходному коду

Недостатки:
- Закрытая платформа
- Высокая стоимость
- Нет поддержки пользовательских расширений
- Сложность миграции на другие платформы

### Выводы по анализу

Проведённый анализ выявляет существенный пробел на рынке: ни одна из существующих платформ не предоставляет полной системы мониторинга с поддержкой пользовательских плагинов, интеграцией Prometheus и централизованным реестром расширений.

Коммерческие платформы (Cilium, Pixie, Falco, Datadog, Dynatrace) фокусируются на узких сценариях использования (сеть, Kubernetes, безопасность) или требуют привязки к облачной инфраструктуре провайдера. Все они предоставляют возможность мониторинга закрытых систем через eBPF, но не позволяют пользователям создавать собственные сборщики метрик для специфических задач.

Таким образом, разработка универсальной eBPF-платформы мониторинга с модульной архитектурой, поддержкой плагинов и экосистемой распространения является обоснованной и востребованной задачей.

## 1.5 Определение границ проекта

В ходе проектирования платформы мониторинга были выявлены границы проекта, которые определяют область ответственности и влияния системы, четко описывая, что включено в её объем, а что исключено. Это обеспечивает ясное понимание целей, задач и ограничений проекта для эффективного планирования, управления и успешного завершения [25].

**Границы функциональности:**

1. Платформа поддерживает только операционную систему Linux с версией ядра 5.8 и выше. Более ранние версии ядра могут не содержать необходимых функций eBPF (например, BTF — BPF Type Format).

2. Система предоставляет мониторинг на уровне ядра и системных вызовов. Мониторинг на уровне приложения (метрики JVM, интерпретаторов языков программирования) не входит в границы проекта и может быть реализован через плагины сторонних разработчиков.

3. Платформа поддерживает экспорт метрик только в формате Prometheus. Интеграция с другими системами мониторинга (InfluxDB, Graphite, Datadog) возможна через шлюзы, разработанные сообществом.

4. Реестр плагинов предоставляет базовый функционал: каталог, загрузку, версионирование, рейтинги. Функции монетизации, платных плагинов и подписок не входят в границы проекта.

5. Веб-интерфейс предоставляет возможности управления плагинами и просмотра метрик в реальном времени. Продвинутая аналитика, машинное обучение и прогнозирование аномалий не реализуются в рамках проекта.

**Границы безопасности:**

6. Платформа использует встроенный верификатор eBPF ядра Linux для проверки безопасности программ. Дополнительные механизмы статического анализа и формальной верификации не входят в границы проекта.

7. Система требует запуска с привилегиями root или capability CAP_BPF для загрузки eBPF-программ. Механизмы работы без привилегий (например, через user namespace) не поддерживаются.

8. Аутентификация и авторизация в веб-интерфейсе реализуются через базовые методы (токены, сессии). Интеграция с корпоративными системами аутентификации (LDAP, SAML, OAuth) не входит в границы проекта.

**Границы производительности:**

9. Платформа ориентирована на мониторинг одиночных узлов (хостов). Агрегация метрик с нескольких узлов, распределенная обработка и кластеризация не реализуются в рамках проекта.

10. Система поддерживает до 100 одновременно загруженных плагинов на одном узле. Сценарии с большим количеством плагинов требуют дополнительного тестирования и оптимизации.

11. Накладные расходы платформы на мониторируемую систему не превышают 5% CPU при типичной нагрузке. Сценарии экстремально высокой нагрузки (миллионы событий в секунду) могут требовать индивидуальной настройки.

**Границы поддержки:**

12. Документация платформы предоставляется на русском и английском языках. Локализация интерфейса на другие языки не входит в границы проекта.

13. Техническая поддержка осуществляется через сообщество (GitHub Issues, форумы). Коммерческая поддержка с SLA не предоставляется в рамках проекта.

14. Платформа распространяется под лицензией MIT. Двойное лицензирование и коммерческие лицензии не предусмотрены.

**Исключения из границ проекта:**

15. Платформа не предоставляет функций оркестрации контейнеров и управления Kubernetes. Интеграция с такими системами возможна через плагины.

16. Система не включает в себя механизмы хранения исторических данных. Долгосрочное хранение и анализ трендов должны реализовываться через интеграцию с Prometheus и Thanos.

17. Платформа не предоставляет готовых дашбордов для конкретных приложений (базы данных, веб-серверы, микросервисы). Такие дашборды могут быть созданы сообществом как плагины.

Определение границ проекта позволяет сфокусировать усилия разработки на ключевой функциональности, избегая распыления ресурсов на второстепенные задачи. Расширение границ возможно в будущих версиях платформы на основе обратной связи от сообщества.

#### Определение функциональных требований

На основе пользовательских требований были определены функциональные требования для выбранной системы.

**Требования к ядру платформы:**

Система должна обеспечивать загрузку и исполнение eBPF-программ в ядре Linux с использованием системных вызовов bpf().

Система должна предоставлять API для управления жизненным циклом плагинов: загрузка, активация, деактивация, выгрузка.

Система должна изолировать выполнение плагинов друг от друга, предотвращая взаимное влияние.

Система должна проверять eBPF-программы через верификатор ядра перед загрузкой.

Система должна собирать метрики производительности самой платформы (потребление памяти, CPU, количество событий).

**Требования к системе плагинов:**

Плагин должен описываться манифестом в формате YAML, содержащим метаданные (название, версия, автор, описание).

Плагин должен включать eBPF-программу в формате ELF с BTF-информацией.

Плагин должен определять выходные метрики с указанием типа (counter, gauge, histogram) и формата.

Плагин должен поддерживать параметры конфигурации с значениями по умолчанию.

Плагин должен указывать требования к совместимости (минимальная версия ядра, зависимости от других плагинов).

Система должна поддерживать горячую загрузку плагинов без перезапуска основного процесса.

Система должна предоставлять механизм отката к предыдущей версии плагина при ошибке загрузки.

**Требования к экспортеру Prometheus:**

Система должна экспортировать метрики в формате Prometheus через HTTP-эндпоинт `/metrics`.

Система должна поддерживать все типы метрик Prometheus: Counter, Gauge, Histogram, Summary.

Система должна добавлять стандартные лейблы к метрикам (hostname, plugin_name, version).

Система должна позволять настраивать порт и путь эндпоинта экспорта.

Система должна поддерживать service discovery через аннотации или файл конфигурации.

**Требования к реестру плагинов:**

Реестр должен предоставлять HTTP API для публикации и загрузки плагинов.

Реестр должен хранить метаданные плагинов: название, версию, описание, автора, рейтинг.

Реестр должен поддерживать версионирование плагинов с возможностью получения конкретной версии.

Реестр должен предоставлять поиск плагинов по названию, тегам, описанию.

Реестр должен позволять пользователям оставлять отзывы и оценки (1–5 звезд).

Реестр должен вести статистику загрузок каждого плагина.

Реестр должен поддерживать категории плагинов (сеть, процессы, файловая система, безопасность).

Реестр должен проверять уникальность названия плагина в пределах одной мажорной версии.

**Требования к веб-интерфейсу:**

Веб-интерфейс должен предоставлять дашборд с обзором состояния платформы и загруженных плагинов.

Веб-интерфейс должен отображать список установленных плагинов с их статусом и метриками.

Веб-интерфейс должен позволять загружать новые плагины из реестра или из файла.

Веб-интерфейс должен предоставлять форму конфигурации параметров плагина.

Веб-интерфейс должен отображать метрики в реальном времени в виде графиков и таблиц.

Веб-интерфейс должен позволять настраивать пороговые значения для уведомлений.

Веб-интерфейс должен предоставлять доступ к логам платформы и плагинов.

Веб-интерфейс должен поддерживать темную и светлую темы оформления.

**Требования к безопасности:**

Система должна требовать аутентификации для доступа к веб-интерфейсу и API управления.

Система должна использовать токены доступа (JWT) для авторизации запросов.

Система должна логировать все операции управления плагинами для аудита.

Система должна ограничивать ресурсы eBPF-программ (максимальное количество инструкций, размер карт).

Система должна предотвращать загрузку плагинов с неподписанным или некорректным кодом.

#### Определение нефункциональных требований

На основе пользовательских и функциональных требований были определены нефункциональные требования для выбранной системы.

**Безопасность:**

Система должна обеспечивать безопасное хранение учетных данных пользователей с использованием хеширования алгоритмом bcrypt или argon2.

Система должна использовать HTTPS для всех внешних соединений с шифрованием TLS 1.2 или выше.

Система должна реализовывать защиту от основных веб-уязвимостей (OWASP Top 10): XSS, CSRF, SQL-инъекции.

Система должна предоставлять механизм ротации токенов доступа и отзыва сессий.

**Производительность:**

Накладные расходы платформы на мониторируемую систему не должны превышать 5% CPU при типичной нагрузке (до 100 000 событий в секунду).

Время загрузки плагина от момента запроса до активации не должно превышать 5 секунд.

Время отклика веб-интерфейса на пользовательские действия не должно превышать 200 мс для 95-го перцентиля.

Система должна поддерживать обработку до 1 миллиона событий eBPF в секунду без потери данных.

Потребление оперативной памяти платформой не должно превышать 500 МБ при 50 загруженных плагинах.

**Надежность:**

Система должна обеспечивать доступность 99,9% в течение месяца (не более 43 минут простоя).

Сбой одного плагина не должен влиять на работу других плагинов и платформы в целом.

Система должна автоматически перезапускать упавшие eBPF-программы в течение 10 секунд.

Система должна сохранять состояние конфигурации при перезапуске процесса платформы.

**Масштабируемость:**

Система должна поддерживать вертикальное масштабирование (увеличение ресурсов узла) без изменения конфигурации.

Архитектура платформы должна позволять горизонтальное масштабирование экспортера метрик.

Система должна поддерживать до 100 одновременно загруженных плагинов на одном узле.

**Устойчивость:**

При потере соединения с реестром платформа должна сохранять работоспособность с локальными плагинами.

При заполнении диска логами система должна автоматически ротировать файлы, сохраняя последние 7 дней.

При некорректной конфигурации плагина система должна откатываться к предыдущей рабочей версии.

**Сохранение данных:**

Конфигурация платформы и плагинов должна сохраняться в файлах формата YAML.

Система должна создавать резервные копии конфигурации при каждом изменении.

История загрузок и обновлений плагинов должна храниться не менее 30 дней.

**Требования к переносимости:**

Платформа должна быть разработана для операционной системы Linux с архитектурой x86_64 и ARM64.

Система должна поставляться в форматах: бинарный файл, Docker-контейнер, пакеты DEB/RPM.

**Требования к установке:**

Необученный пользователь должен суметь выполнить начальную установку платформы в среднем за 15 минут по инструкции.

При установке обновленной версии платформы должна сохраняться конфигурация и установленные плагины.

Система должна предоставлять скрипты автоматической установки для популярных дистрибутивов Linux.

### 1.6.2 Ограничения проекта

Определением требований в проекте было четко определено, что система должна делать. Теперь можно установить ограничения проекта, чтобы определить границы и рамки, в которых эти требования могут быть успешно реализованы:

Платформа должна поддерживать только ядро Linux версии 5.8 и выше. Более ранние версии могут не содержать необходимых функций eBPF (BTF, BPF_LINK).

Для работы платформы требуются привилегии root или capability CAP_BPF, CAP_PERFMON, CAP_NET_ADMIN для загрузки eBPF-программ.

Максимальное количество одновременно загруженных плагинов на одном узле ограничено 100 для обеспечения стабильности системы.

Размер одного плагина не должен превышать 10 МБ (включая eBPF-программу, манифест и ресурсы).

eBPF-программа в составе плагина не должна превышать 1 миллиона инструкций (ограничение верификатора ядра).

Все eBPF-программы должны проходить верификацию ядра Linux. Программы, не прошедшие проверку, не могут быть загружены.

Платформа не поддерживает работу в средах с усиленной безопасностью ядра (lockdown mode) без дополнительной настройки.

Реестр плагинов хранит только метаданные и бинарные файлы плагинов. Исходный код eBPF-программ не требуется публиковать.

Веб-интерфейс поддерживает только современные браузеры: Chrome 90+, Firefox 88+, Safari 14+, Edge 90+.

Для экспорта метрик требуется отдельный сервер Prometheus. Платформа не включает в себя систему хранения временных рядов.

Все текстовые данные интерфейса хранятся в виде JSON-файлов локализации. Поддержка RTL-языков (арабский, иврит) не реализуется.

Для разработки плагинов требуется инструментальная цепочка eBPF: clang, LLVM, libbpf. Платформа не предоставляет встроенную среду разработки.

Ограничения проекта являются основой для установки требований к программному обеспечению, техническому обеспечению, информационному обеспечению, документации на программное решение, а также требованиям к эксплуатации программного решения, обеспечивая единое и согласованное руководство в рамках проекта.

### 1.6.3 Требования к программному обеспечению

Для обеспечения эффективного и надежного функционирования платформы мониторинга, удовлетворения потребностей пользователей, обеспечения безопасности данных и соблюдения стандартов разработки, были определены требования к программному обеспечению [27]:

**Backend (ядро платформы):**

Серверная часть платформы должна быть разработана на языке программирования Go версии 1.21 или выше.

Для работы с eBPF должна использоваться библиотека libbpf через Go bindings (github.com/cilium/ebpf).

Система должна использовать структурированное логирование через библиотеку zap или logrus.

Конфигурация должна загружаться из файлов формата YAML с использованием библиотеки viper.

**Frontend (веб-интерфейс):**

Веб-интерфейс должен быть разработан с использованием фреймворка Svelte версии 4 или выше.

Для управления состоянием приложения должна использоваться библиотека store Svelte или nanostores.

Для визуализации метрик должны использоваться библиотеки: Chart.js или D3.js для графиков.

Для HTTP-запросов к API должна использоваться встроенная функция fetch.

**Реестр плагинов:**

Сервер реестра должен быть разработан на языке Go с использованием фреймворка Gin или Echo.

Для хранения метаданных плагинов должна использоваться база данных PostgreSQL версии 14 или выше.

Для кэширования часто запрашиваемых данных должен использоваться Redis.

**Инструменты разработки:**

Для компиляции eBPF-программ должен использоваться clang версии 14 или выше с поддержкой BTF.

Для сборки проекта должна использоваться система Make с декларативными Makefile.

Для управления зависимостями Go должен использоваться Go modules.

Для управления зависимостями Node.js должен использоваться npm или pnpm.

**Тестирование:**

Модульные тесты должны писаться с использованием встроенного пакета testing для Go.

Интеграционные тесты должны использовать библиотеку testify для утверждений.

E2E-тесты веб-интерфейса должны использовать Playwright или Cypress.

Тесты eBPF-программ должны использовать среду bpftrace или custom test loader.

**Контейнеризация:**

Платформа должна поставляться в Docker-контейнере на базе образа Alpine Linux или distroless.

Для оркестрации контейнеров в Kubernetes должны предоставляться манифесты Deployment, Service, ConfigMap.

### 1.6.7 Требования к эксплуатации программного решения

Для обеспечения устойчивой и эффективной работы платформы мониторинга после её внедрения, для минимизации рисков сбоев и максимизации удобства пользователей, были определены требования к эксплуатации программного решения [28]:

**Поддержка платформ:**

Платформа должна быть совместима с актуальными LTS-версиями дистрибутивов Linux: Ubuntu 20.04/22.04, Debian 11/12, RHEL 8/9, CentOS Stream 9.

Платформа должна регулярно обновляться для поддержания совместимости с новыми версиями ядра Linux (до 6 месяцев после выхода стабильной версии ядра).

**Доступность и надежность:**

Платформа должна быть доступна для использования 24/7 за исключением временных окон планового обслуживания (не более 4 часов в месяц).

Система должна предоставлять health-check эндпоинт для мониторинга доступности через `/healthz`.

Система должна автоматически перезапускаться при сбое через systemd или контейнерный оркестратор.

**Мониторинг и логирование:**

Платформа должна экспортировать метрики собственной производительности через отдельный эндпоинт.

Система должна логировать все критические события: загрузка/выгрузка плагинов, ошибки верификации, превышение лимитов ресурсов.

Логи должны быть структурированы в формате JSON для интеграции с системами централизованного логирования (ELK, Loki).

**Обновления и патчи:**

Регулярные обновления платформы для устранения уязвимостей безопасности должны выпускаться в течение 7 дней после публикации CVE.

Обновления версий платформы должны следовать семантическому версионированию (SemVer).

Процесс обновления не должен требовать простоя системы более 1 минуты.

**Резервное копирование и восстановление:**

Платформа должна предоставлять утилиту для создания резервных копий конфигурации: `ebpf-monitor backup`.

Процедура восстановления из резервной копии должна занимать не более 15 минут.

Резервные копии должны храниться локально и опционально загружаться в облачное хранилище (S3, GCS).

**Безопасность:**

Платформа должна требовать регулярной ротации токенов доступа (не реже одного раза в 90 дней).

Система должна уведомлять администратора о подозрительной активности: множественные неудачные попытки аутентификации, загрузка плагинов из непроверенных источников.

Обновления безопасности ядра Linux должны применяться в течение 30 дней после выхода для обеспечения совместимости eBPF.

**Поддержка сообщества:**

Документация должна быть доступна онлайн с поиском по содержанию.

Пользователи должны иметь возможность сообщать об ошибках через GitHub Issues.

Разработчики должны предоставлять ответы на критические вопросы в течение 5 рабочих дней.

**Обучение и адаптация:**

Платформа должна предоставлять интерактивные примеры конфигурации для типовых сценариев.

Должны быть доступны видео-туториалы по установке, настройке и созданию плагинов.

Сообщество должно поддерживать форум или чат (Discord, Slack) для взаимопомощи пользователей.
