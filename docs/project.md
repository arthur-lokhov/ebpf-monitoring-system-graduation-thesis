# 2 ПРОЕКТНЫЙ РАЗДЕЛ

## 2.1 Модель системы "TO-BE"

Для описания выполняемых системой функций и представления системы в виде иерархии диаграмм [29], построим функциональную модель в нотации IDEF0, используя входные и выходные данные, определенные в разделе 1.2.

Модель в нотации IDEF0 представляет собой дерево иерархически упорядоченных и взаимосвязанных диаграмм. Каждая диаграмма является единицей описания системы и располагается на отдельном листе [12]. Построенные диаграммы отображены на рисунках 2.1–2.6.

Построенная модель IDEF0 включает в себя детальное моделирование процессов внутри системы, что в свою очередь помогает установить архитектуру будущего программного решения.

Ключевым аспектом модели "TO-BE" является событийно-ориентированная архитектура: в отличие от модели "AS-IS", где метрики собираются через циклический опрос, eBPF-программы активируются только при наступлении событий ядра. Это обеспечивает:

- Минимальные накладные расходы на мониторируемую систему;
- Мгновенное обнаружение событий без задержек;
- Возможность мониторинга закрытых систем («черных ящиков») через отслеживание системных вызовов и событий ядра без доступа к исходному коду приложений.

## 2.2 Архитектура программного решения

### 2.2.1 Архитектурная диаграмма

Начнем построение архитектуры программного решения с построения архитектурной диаграммы eBPF-платформы мониторинга в соответствии с моделью C4.

Архитектурная диаграмма служит для визуализации и структурирования архитектуры системы, позволяя четко представить взаимосвязи между ее ключевыми компонентами, модулями и интерфейсами, что облегчает понимание, проектирование и обсуждение архитектурных решений в ходе разработки программного продукта или системы [30].

### 2.2.2 Выбор и обоснование архитектурного паттерна проектирования кода приложения

Для разработки backend-части eBPF-платформы мониторинга был выбран архитектурный паттерн **Clean Architecture** (Чистая архитектура) с элементами **Event-Driven Architecture** (Событийно-ориентированная архитектура).

Данный выбор обусловлен следующими преимуществами [31]:

**1. Разделение ответственностей.**

Clean Architecture разделяет систему на независимые уровни:
- **Entities (Сущности)** — бизнес-правила платформы (модели плагинов, метрик, пользователей);
- **Use Cases (Сценарии использования)** — конкретные операции (загрузка плагина, экспорт метрик, валидация eBPF-кода);
- **Interface Adapters (Адаптеры интерфейсов)** — преобразование данных между уровнями (HTTP-обработчики, сериализаторы);
- **Frameworks & Drivers (Фреймворки и драйверы)** — внешние зависимости (libbpf, база данных, HTTP-сервер).

Такое разделение позволяет изменять реализацию одного уровня без влияния на другие. Например, замена базы данных PostgreSQL на MongoDB не затронет бизнес-логику платформы.

**2. Независимость от фреймворков.**

Ядро платформы не зависит от конкретных библиотек и фреймворков. Libbpf используется только на внешнем уровне, что позволяет при необходимости заменить его на другую библиотеку (например, gobpf) без изменения бизнес-логики.

**3. Тестируемость.**

Четкое разделение уровней упрощает написание модульных тестов. Use Cases могут быть протестированы изолированно с использованием mock-объектов для внешних зависимостей.

**4. Событийно-ориентированная архитектура для обработки метрик.**

Для обработки событий eBPF используется Event-Driven Architecture:
- eBPF-программы генерируют события при наступлении условий в ядре;
- Event Bus распределяет события между обработчиками;
- Обработчики агрегируют события и формируют метрики.

Этот подход обеспечивает:
- **Асинхронность** — обработка событий не блокирует основной поток;
- **Масштабируемость** — возможность добавлять новые обработчики без изменения существующего кода;
- **Гибкость** — поддержка динамической загрузки обработчиков через плагины.

**5. Расширяемость через плагины.**

Архитектура поддерживает механизм плагинов:
- Плагины реализуют общий интерфейс (Plugin Interface);
- Plugin Manager загружает и активирует плагины динамически;
- Новые плагины могут быть добавлены без модификации ядра платформы.

**6. Поддержка мониторинга «черных ящиков».**

Событийно-ориентированная архитектура позволяет собирать метрики о работе любых приложений через eBPF:
- eBPF-программы отслеживают системные вызовы и события ядра;
- Данные агрегируются независимо от внутренней структуры мониторируемого приложения;
- Метрики экспортируются в стандартном формате Prometheus.

Таким образом, комбинация Clean Architecture и Event-Driven Architecture обеспечивает модульность, тестируемость и расширяемость платформы, что критически важно для системы с поддержкой пользовательских плагинов.

### 2.2.3 Масштабируемость системы

После построения архитектурной диаграммы определим масштабируемость системы, поскольку архитектурная диаграмма визуализирует внутреннюю структуру и взаимосвязи компонентов, что позволяет более эффективно проектировать и планировать систему с учетом потенциала её масштабируемости.

Определение масштабируемости системы необходимо для предвидения и оценки её способности эффективно расти и адаптироваться к увеличению объема данных, трафика или пользовательской нагрузки, что обеспечивает устойчивость и успешное функционирование системы в условиях изменяющихся требований и объемов использования [32].

eBPF-платформа мониторинга позволяет обеспечить управление растущей нагрузкой на систему, обеспечивая производительность и стабильность системы, удовлетворяя потребности пользователей.

**Вертикальное масштабирование:**

При росте числа событий eBPF или количества плагинов можно увеличить вычислительную мощность сервера:
- Добавление CPU-ядер для параллельной обработки событий от нескольких eBPF-программ;
- Увеличение оперативной памяти для хранения большего количества карт (maps) eBPF и буферов событий;
- Использование SSD для ускорения записи логов и чтения конфигураций плагинов.

**Горизонтальное масштабирование:**

Для горизонтального масштабирования будут использоваться следующие стратегии:
- **Кластеризация экспортеров метрик** — несколько экземпляров Metrics Exporter могут работать параллельно, каждый обслуживая свой набор плагинов;
- **Шардирование реестра плагинов** — база данных реестра может быть распределена между несколькими узлами для обработки большего количества запросов;
- **Распределенный кэш** — Redis Cluster для кэширования часто запрашиваемых метаданных плагинов и метрик.

**Оптимизация производительности:**

Помимо стратегий вертикального и горизонтального масштабирования может быть применена оптимизация, реализуемая при помощи:
- Использования более эффективных алгоритмов агрегации событий (например, скользящие окна вместо полных пересчетов);
- Добавления индексов в базу данных реестра для ускорения поиска плагинов;
- Применения кольцевых буферов (ring buffers) eBPF для эффективной передачи событий из ядра в пользовательское пространство;
- Батчинга событий — группировки нескольких событий перед обработкой для снижения накладных расходов.

**Метрики для тестирования масштабируемости:**

Для тестирования масштабируемости системы будут использоваться следующие характеристики:

1. **Производительность:**
   - Время отклика (Response Time): Измерение времени, требующегося для обработки событий eBPF и экспорта метрик. Это позволит оценить, как быстро система реагирует на события при различных нагрузках.

2. **Пропускная способность (Throughput):**
   - Измерение количества событий eBPF, которые система может обработать в единицу времени. Это позволит определить, сколько событий система способна обслуживать при заданной нагрузке.

3. **Стабильность:**
   - Тестирование стабильности системы при увеличенной нагрузке. Проверка, сохраняется ли стабильность системы при росте количества плагинов и событий.

4. **Масштабируемость:**
   - Оценка способности системы масштабироваться. Измерение, как система реагирует на добавление ресурсов (CPU, память) или узлов (горизонтальное масштабирование).

5. **Загрузочное тестирование (Load Testing):**
   - Проведение тестов при максимально возможной нагрузке (миллионы событий в секунду) для определения, как система будет работать в условиях пика активности.

6. **Тестирование стабильности базы данных:**
   - Проверка, как база данных реестра реагирует на увеличение запросов. Мониторинг производительности базы данных и оценка ее способности обработки большого объема метаданных плагинов.

7. **Тестирование отказоустойчивости:**
   - Симуляция отказов компонентов (eBPF-программы, экспортер, реестр) для определения, как система реагирует на сбои и восстанавливается после них.

**Ограничения масштабируемости:**

- Максимальное количество одновременно загруженных плагинов на одном узле ограничено 100 для обеспечения стабильности;
- eBPF-программы исполняются в ядре, поэтому их количество и сложность влияют на производительность всей системы;
- Верификатор eBPF накладывает ограничения на сложность программ (максимальное количество инструкций).

## 2.6 Выбор и обоснование стека технологий

### 2.6.1 Backend (Go)

Для разработки серверной части платформы был выбран язык программирования **Go** версии 1.21 или выше.

**Обоснование выбора:**

1. **Производительность.** Go компилируется в нативный машинный код, обеспечивая высокую производительность, сравнимую с C/C++. Это критически важно для обработки миллионов событий eBPF в секунду [34].

2. **Встроенная поддержка конкурентности.** Модель горутин и каналов Go упрощает разработку асинхронных систем, что необходимо для обработки событий от нескольких eBPF-программ параллельно.

3. **Библиотеки для работы с eBPF.** Экосистема Go предоставляет зрелые библиотеки для работы с eBPF: `github.com/cilium/ebpf` (официальная библиотека Cilium), `github.com/iovisor/gobpf`.

4. **Минимальные накладные расходы.** Go не требует виртуальной машины или сборщика мусора с большими паузами, что важно для систем реального времени.

5. **Статическая компоновка.** Go-программы компилируются в единый бинарный файл без внешних зависимостей, что упрощает развертывание.

6. **Широкое применение в инфраструктуре.** Go используется для разработки Kubernetes, Docker, Prometheus, etcd — это обеспечивает хорошую интеграцию с экосистемой cloud-native инструментов.

### 2.6.2 Frontend (Svelte)

Для разработки веб-интерфейса был выбран фреймворк **Svelte** версии 4 или выше.

**Обоснование выбора:**

1. **Компиляция вместо runtime.** Svelte компилирует компоненты в оптимизированный JavaScript на этапе сборки, что устраняет накладные расходы виртуального DOM и обеспечивает высокую производительность [35].

2. **Минимальный размер бандла.** Svelte генерирует значительно меньше кода по сравнению с React и Vue, что ускоряет загрузку интерфейса.

3. **Встроенное управление состоянием.** Svelte stores предоставляют простой механизм реактивного управления состоянием без необходимости в дополнительных библиотеках.

4. **Простота изучения.** Svelte имеет более низкий порог входа по сравнению с React и Vue, что упрощает разработку и поддержку кода.

5. **Отсутствие зависимостей runtime.** Svelte не требует подключения тяжелых библиотек в браузер, что снижает размер приложения.

### 2.6.3 eBPF (libbpf)

Для работы с eBPF была выбрана библиотека **libbpf**.

**Обоснование выбора:**

1. **Официальная поддержка.** libbpf является частью ядра Linux и поддерживается сообществом eBPF [36].

2. **BTF (BPF Type Format).** libbpf предоставляет полную поддержку BTF для отладки и интроспекции eBPF-программ.

3. **Минимальные накладные расходы.** libbpf — это низкоуровневая библиотека на C, обеспечивающая прямой доступ к системным вызовам bpf().

4. **Go bindings.** Библиотека `github.com/cilium/ebpf` предоставляет идиоматичные Go bindings для libbpf.

5. **Стабильность API.** libbpf гарантирует стабильность API между версиями, что упрощает поддержку кода.

### 2.6.4 База данных (PostgreSQL)

Для хранения метаданных реестра была выбрана СУБД **PostgreSQL** версии 14 или выше.

**Обоснование выбора:**

1. **Надежность.** PostgreSQL обеспечивает ACID-транзакции и целостность данных, что критически важно для хранения метаданных плагинов и отзывов [37].

2. **Расширяемость.** PostgreSQL поддерживает пользовательские типы данных, индексы (GIN, GiST) и расширения.

3. **JSONB.** Поддержка JSONB позволяет хранить гибкие структуры данных (манифесты плагинов) с возможностью индексации.

4. **Репликация.** Встроенная поддержка репликации и кластеризации для обеспечения высокой доступности.

5. **Экосистема.** Богатая экосистема инструментов (миграции, ORM, мониторинг) упрощает разработку и эксплуатацию.

### 2.6.5 Хранилище плагинов (File System / S3)

Для хранения файлов плагинов (ELF-бинарники, манифесты, архивы) предусмотрены два варианта реализации:

**Вариант 1: Локальная файловая система**

- Простота развертывания (не требует дополнительных сервисов)
- Подходит для одиночных узлов и тестовых сред
- Ограниченная масштабируемость

**Вариант 2: Объектное хранилище (S3-совместимое)**

- Минимальная стоимость хранения
- Бесконечная масштабируемость
- Географическая репликация
- Интеграция с CDN для быстрой загрузки

**Рекомендация:** Для продакшн-развертываний рекомендуется использовать S3-совместимое хранилище (AWS S3, MinIO, Cloudflare R2). Для локальной разработки и тестирования достаточно файловой системы.

## 2.7 Обоснование выбора хранилища для плагинов

### 2.7.1 Стоит ли интегрировать S3 как хранилище файлов с плагинами?

Интеграция S3 (или S3-совместимого хранилища) для хранения файлов плагинов является **рекомендуемым решением** для продакшн-развертываний по следующим причинам:

**Преимущества S3:**

1. **Масштабируемость.** S3 предоставляет практически неограниченное хранилище, что критически важно при росте количества плагинов и их версий.

2. **Доступность.** S3 обеспечивает высокую доступность (99,99% SLA у AWS S3) и долговечность данных (99,999999999%).

3. **Географическое распределение.** Плагины могут храниться в регионах, близких к пользователям, что снижает задержки при загрузке.

4. **Версионирование.** S3 поддерживает версионирование объектов, что упрощает управление версиями плагинов и откат при необходимости.

5. **Безопасность.** S3 предоставляет встроенные механизмы шифрования (at-rest и in-transit), управления доступом (IAM policies) и аудита.

6. **Экономическая эффективность.** Оплата только за фактически используемое место и трафик. Для небольших проектов стоимость минимальна.

7. **Экосистема.** S3-совместимые решения (MinIO, Ceph, Cloudflare R2) позволяют развернуть хранилище on-premise при необходимости.

**Недостатки S3:**

1. **Зависимость от внешнего сервиса.** При использовании облачного S3 возникает зависимость от провайдера.

2. **Задержки сети.** Доступ к файлам через сеть медленнее, чем к локальной файловой системе.

3. **Стоимость трафика.** При больших объемах загрузок стоимость исходящего трафика может быть значительной.

**Рекомендация:**

Для дипломного проекта и начальных стадий разработки рекомендуется использовать **гибридный подход**:
- **Разработка/тестирование:** локальная файловая система (`/var/lib/ebpf-monitor/plugins/`)
- **Продакшн:** S3-совместимое хранилище (MinIO для on-premise или AWS S3 для облака)

Такой подход позволяет:
- Начать разработку без дополнительных зависимостей
- Легко мигрировать на S3 при масштабировании
- Поддерживать оба режима через абстракцию хранилища (Storage Interface)

**Абстракция хранилища:**

```go
type PluginStorage interface {
    Upload(ctx context.Context, pluginID string, data io.Reader) error
    Download(ctx context.Context, pluginID string) (io.ReadCloser, error)
    Delete(ctx context.Context, pluginID string) error
    Exists(ctx context.Context, pluginID string) (bool, error)
}

// Реализация для файловой системы
type FileSystemStorage struct {
    basePath string
}

// Реализация для S3
type S3Storage struct {
    client *s3.Client
    bucket string
}
```

Такая архитектура позволяет переключаться между хранилищами через конфигурацию без изменения кода платформы.

## ЗАКЛЮЧЕНИЕ

В ходе выполнения проектного раздела было проведено всестороннее проектирование eBPF-платформы мониторинга. Работа включала построение функциональной модели системы в нотации IDEF0 с соблюдением правил декомпозиции (входы, выходы, механизмы, вызовы наследуются с уровня на уровень), разработку архитектуры в соответствии с моделью C4 (диаграмма контекста, контейнеров, компонентов), определение дерева функций и логики работы программного решения.

Была спроектирована модель системы "TO-BE", которая демонстрирует переход от традиционного polling-мониторинга к событийно-ориентированной архитектуре на основе eBPF. Ключевым преимуществом новой модели является возможность мониторинга закрытых систем («черных ящиков») без доступа к исходному коду приложений.

Архитектура платформы была спроектирована с использованием паттерна Clean Architecture с элементами Event-Driven Architecture, что обеспечивает модульность, тестируемость и расширяемость системы. Были определены масштабируемость системы и стратегии вертикального и горизонтального масштабирования.

Была разработана логическая модель базы данных для реестра плагинов, определены роли пользователей и матрица доступа, составлен словарь данных.

Выбранный стек технологий (Go, Svelte, libbpf, PostgreSQL) обоснован с точки зрения производительности, экосистемы и соответствия требованиям проекта. Для хранения файлов плагинов рекомендовано использование S3-совместимого хранилища с поддержкой гибридного режима (файловая система для разработки, S3 для продакшн).

Разработанные проектные решения могут служить надежной основой для дальнейшей реализации полнофункциональной eBPF-платформы мониторинга с поддержкой плагинов, экспортом в Prometheus и централизованным реестром расширений.
