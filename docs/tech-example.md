ВВЕДЕНИЕ

В современном мире цифровые технологии играют ключевую роль в жизни человека, облегчая хранение, обработку и передачу информации. Развитие мобильных приложений позволяет пользователям не только фиксировать важные события, но и структурировать воспоминания, создавая персонализированные цифровые архивы.
Однако существующие решения зачастую ограничены либо только фото и видеоконтентом, либо недостаточным уровнем взаимодействия между пользователями. В связи с этим возникает необходимость разработки универсального инструмента, который объединяет мультимедийные данные и позволяет пользователям эффективно управлять своими воспоминаниями.
Актуальность выбранной темы обусловлена растущей потребностью пользователей в удобных и функциональных средствах для хранения и организации мультимедийного контента. В условиях цифровизации и постоянного увеличения объема информации важно не только сохранять данные, но и предоставлять пользователям удобные механизмы их структурирования, поиска и совместного использования. Разработка мобильного приложения, интегрирующего текстовые, фото-, аудио- и видеозаписи, позволит пользователям создавать полноценные цифровые дневники, тем самым удовлетворяя их потребности в сохранении личных и культурных ценностей.
Целью данной практической работы является проектирование мобильного приложения, позволяющего пользователям создавать, редактировать, организовывать и делиться мультимедийными воспоминаниями, а также взаимодействовать с другими пользователями через временные выставки и систему подписок.
1 ТЕХНОЛОГИЧЕСКИЙ РАЗДЕЛ
1.1 Выбор средств разработки
Для разработки приложения «Memory» будут использованы следующие средства разработки, которые обеспечат удобство разработки, поддержку актуальных технологий и удовлетворение функциональных требований.
1) Язык программирования:
− Kotlin — основной язык для разработки Android-приложений.
Обоснование выбора:
− Официально поддерживает Google и является предпочтительным для разработки приложений Android. Kotlin, разработанный специально для Android, содержит инструменты, которые упрощают составление надежного кода, делают его более понятным и безопасным.
2) Среда разработки:
− Android Studio.
Обоснование выбора:
− Android Studio: официальная среда разработки для Android-приложений. Она предоставляет мощные инструменты для создания, отладки и тестирования приложений. Она предоставляет удобный интерфейс и встроенные инструменты для написания, тестирования и отладки кода на Kotlin. Также в Android Studio встроены средства эмуляции, приложение для тестирования в разных версиях и модели устройств, которые ускоряют процесс разработки и помогают заранее выявить ошибки.
3) Фреймворки и библиотеки:
− Firebase: В данном случае Firebase используется как облачная база данных и хранилище, а также для аутентификации пользователей и протокола регистрации. Firebase оптимизирует работу с данными, синхронизирует их и обеспечивает безопасность;
− SQLite: для локального хранения данных;
− FFmpeg: это библиотека для сжатия и преобразования медиафайлов (видео и аудио). Она позволяет уменьшить размер файлов перед загрузкой на сервер, что оптимизирует использование места и помогает сократить время загрузки для пользователей;
− Jetpack Compose: современный инструмент для разработки пользовательских интерфейсов на Android. Он упрощает создание адаптивных и интерактивных интерфейсов с помощью декларативного подхода, позволяя разработчикам более эффективно управлять UI-компонентами и их состоянием. Jetpack Compose обеспечивает гибкость и упрощает интеграцию с другими библиотеками и фреймворками;
− Room: для работы с локальной базой данных SQLite, базовой операции CRUD и синхронизации данных с облаком;
− Glide: это библиотека для работы с изображениями, которая поддерживает Android. Она помогает легко загружать изображения в приложение, управлять их кешированием и сжимать их для экономии места.
4) Средства контроля версий:
− Git: для контроля версий и отслеживания изменений в коде;
− GitHub: позволяет безопасно хранить и управлять версиями кода, обеспечивает совместную работу и сохранение всех изменений. Это особенно важно для разработки приложений с возможными улучшениями и расширениями в будущем.
5) Инструменты для тестирования:
− JUnit: используется для модульного тестирования, поэтому позволяет проверить логику кода на корректность;
− Espresso: помогает протестировать пользовательский интерфейс, чтобы убедиться, что все функции работают стабильно и работают на различных устройствах.
1.2 Физическая модель базы данных
Физическая модель данных в контексте СУБД (системы управления базами данных) представляет собой детализированное описание, как данные будут храниться, и организовываться внутри конкретной базы данных, учитывая спецификации и характеристики конкретной СУБД. Она определяет структуру таблиц, индексы, типы данных, размеры полей, механизмы хранения, а также правила для обеспечения целостности данных и оптимизации запросов.
Рисунок 1.1 демонстрирует физическую модель базы данных.
Рисунок 1.1 – Физическая модель

Все внешние ключи (FK) обеспечивают целостность данных.
NN (NOT NULL) – обязательные поля.
Физическая модель базы данных включает четыре таблицы:
1) Users (Пользователи):
− users.id к exhibits.user_id (Один пользователь может иметь много экспонатов);
− users.id к albums.user_id (Один пользователь может иметь много альбомов);
− users.id к comments.user_id (Один пользователь может оставить много комментариев).
2) Exhibits (Экспонаты):
− exhibits.id -> album_exhibits.exhibit_id (Один экспонат может быть в нескольких альбомах);
− exhibits.id -> comments.exhibit_id (Один экспонат может иметь несколько комментариев);
− exhibits.id -> media_files.exhibit_id (Один экспонат может иметь несколько медиафайлов);
− exhibits.user_id -> users.id (Экспонат принадлежит пользователю).
3) Albums (Альбомы):
− albums.id к album_exhibits.album_id (Один альбом может содержать много экспонатов);
− albums.user_id -> users.id (Альбом принадлежит пользователю);
4) Album_Exhibits (Экспонаты в альбоме):
− album_exhibits.album_id к albums.id (Связь с альбомом);
− album_exhibits.exhibit_id к exhibits.id (Связь с экспонатом).
5) Notifications (Уведомления):
− notifications.user_id к users.id (Уведомление для конкретного пользователя).
6) Media_Files (Медиафайлы):
− media_files.exhibit_id к exhibits.id (Медиафайл принадлежит экспонату).
7) Comments (Комментарии):
− comments.exhibit_id к exhibits.id (Комментарий относится к экспонату);
− comments.user_id к users.id (Комментарий написан пользователем).
1.3 Схема развертывания аппаратного и программного обеспечения
Аппаратное обеспечение:
1) Серверная часть:
⎯ Firebase Database – облачный сервер,
⎯ Балансировщик нагрузки для масштабируемости.
2) База данных:
⎯ MySQL – сервер базы данных,
⎯ репликация базы данных для отказоустойчивости.
3) Клиентские устройства:
⎯ смартфоны (Android),
⎯ подключение через интернет (Wi-Fi, мобильная сеть).
Программное обеспечение:
1) Серверная часть:
⎯ Firebase Authentication – аутентификация,
⎯ Firebase Storage – файловое хранилище.
2) База данных:
⎯ MySQL – реляционная база данных.
3) Клиентская часть:
⎯ разработка на Kotlin,
⎯ UI/UX дизайн с Material Design,
⎯ взаимодействие с сервером через HTTP (REST/GraphQL)
Взаимодействие компонентов:
1. Пользователь открывает мобильное приложение.
2. Приложение отправляет запрос на сервер (например, выложить экспонат).
3. Серверная часть обрабатывает запрос, взаимодействует с базой данных.
4. База данных возвращает данные серверу.
5. Сервер отправляет данные обратно в мобильное приложение.
6. Приложение отображает данные пользователю.

ЗАКЛЮЧЕНИЕ

В ходе работы над проектом было спроектировано мобильное приложение Memory, которое предоставляет пользователям возможность создавать, редактировать и организовывать мультимедийные воспоминания. Приложение интегрирует текстовые, фото, аудио и видеозаписи, создавая универсальный инструмент для управления контентом.
Актуальность проекта обусловлена растущей потребностью пользователей в удобных инструментах для хранения, структурирования и совместного использования мультимедийного контента.
Анализ аналогичных решений позволил выявить их сильные и слабые стороны, что дало возможность учесть полученные выводы при проектировании системы. В ходе работы были определены ключевые функциональные и нефункциональные требования, спроектирована архитектура приложения.
Тестирование приложения подтвердило его работоспособность, удобство использования и надёжность системы. Внедрение механизма социального взаимодействия делает Memory не просто инструментом для хранения воспоминаний, а полноценной платформой для общения и обмена важными моментами жизни.
Практическая значимость проекта заключается в создании удобной и функциональной социальной сети, позволяющей пользователям не только сохранять важные события, но и делиться ими с другими людьми. Memory способствует укреплению социальных связей, объединяя пользователей на основе их воспоминаний и опыта.
